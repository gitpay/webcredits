Purpose
Linked data based Credits.

Abstract

We present a system of credits powered by linked data and the web. A web credit is defined as a credit between two URIs of a given amount and currency, at a given time. It can perform simple workflows on its own, such as IOUs between entities, but is intended largely as a building block for use in larger workflows.

Keywords

  • Linked Data

Introduction

We present credit based system for linked data and the world wide web. The model is simple and allows for a credit to be issued between two URIs, for an amount and in a specific currency, at a given time. It is intended to be part of a larger workflow, however can be used as stand alone for simple cases. We also present a simple server that can be used to process, validate and aggregate accumulated credits.

Non-Functional Requirements

Decentralization

As much of the system as possible should be decentralized. This includes, identity, accounts, credits, wallets and messaging. This is achieved by giving anything of note a URI. That allows any domain to host any aspect of the system.

Accessibility

Accessiblity and ARIA guidelines are a goal of the project, so that as many people and possible can use wallets

Web Standards

As far as possible this project aims to reuse web standards and work done by the W3C. The main items followed are W3C Design Issues, WebID, Linked Data, Linked Data Platform, HTML5, RESTful principles and Websockets.

Security

Work is done on security, in particular the tools used are Web Access Control, and TLS. More tools will be added over time.

Conceptual Design

The system deals with discovery of information, from user, to workspace, to appication specific data storage. It deals with finding where to PUT a new transaction.

Discovery

Discovery is the act of following a pointer or string in order to establish more information. The primary technicque, for discovery, is "Follow You Nose", dereferencing the identifier, usually using HTTP and finding out more information. As a temporary or fallback measure well known locations can be used, or search engines, to augment follow your nose.

Components

As far as possible this project aims to reuse web standards and work done by the W3C. The main items followed are SoLiD, W3C Design Issues, WebID, Linked Data, Linked Data Platform, HTML5, RESTful principles and Websockets

WebID

WebID will be used for Identity. A WebID is an HTTP URI that denotes an Agent. Hash URIs are preferred. The system can later be exteneded to include any URI such as email, hashes, tel: and devices.

LDP

Linked Data Principles will be used throughout this work, and Linked Data Platform as a commodity back end.

Web Credits

The web credits ontology will be used as a simple way to track credits between URIs.

W3C Web Payments

The work by the W3C web payments community group will be leveraged as much as possible.

Storage

We use a web powered server written in nodejs, to store the initial credits, using HTTP PUT, which is part of the linked data platform. Once the new credit is put in an agreed container, anothe process monitors the container and will validate, add an UUID to the credit and send it downsteam for archiving and indexing. For our work we are using the GOLD linked data server.

The format of the transaction is: 

{"source":"http://bitbots.org/workbot#this",
                                "amount":25,
                                "currency":"https://w3id.org/cc#bit",
                                "destination":"http://melvincarvalho.com/#me",
                                "timestamp":"2015-03-28T09:55:29.803Z",
                                "context":null,
                                "description":"work"}

An update to a well known file, ",meta" is also sent by the client in order to let the process know there is a new transaction.

The server will then vaildate that all required fields are filled in, check that the balance is above the amount needed, and add a UUID. It will also send the message downstream, deleting the existing tx, using HTTP DELETE. In this case we use a single table sqlite database to index processed transactions. This works well on small scale, but due to table locking a more robust database, perhaps such as redis, would be used for more transactions.

When the process is finished, it again updates ",meta" in order to let the client know there was an update.

Conclusions

In this article, have described a simple method for creating web credits, validating them and updating resources in realtime. Future work will deal with scalability to handle trillions of transactions, and integrations between systems.
